## 1. Аутентификация и управление аккаунтом

### UC-1.1: Регистрация нового пользователя

**Описание:** Пользователь регистрирует новый аккаунт в системе.

**Актор:** Новый пользователь

**Предусловия:**

- Пользователь не авторизован
- Пользователь находится на странице регистрации

**Основной поток:**

1. Пользователь открывает страницу регистрации
2. Пользователь заполняет форму:
   - Email (обязательное поле)
   - Пароль (обязательное поле, минимальная длина 8 символов)
   - Подтверждение пароля (обязательное поле, должно совпадать с паролем)
   - Имя (обязательное поле)
   - Фамилия (обязательное поле)
3. Система валидирует введенные данные:
   - Email должен быть корректным форматом
   - Пароль должен соответствовать требованиям безопасности
   - Подтверждение пароля должно совпадать с паролем
   - Все обязательные поля должны быть заполнены
4. Пользователь нажимает кнопку "Зарегистрироваться"
5. Система создает новый аккаунт
6. Система отправляет JWT токен пользователю
7. Пользователь автоматически авторизуется
8. Пользователь перенаправляется на Dashboard

**Альтернативные потоки:**

**A1:** Email уже существует

- Система отображает ошибку "Email уже зарегистрирован"
- Пользователь может ввести другой email или перейти на страницу входа

**A2:** Пароль не соответствует требованиям

- Система отображает ошибку валидации пароля
- Пользователь может ввести новый пароль

**A3:** Пароли не совпадают

- Система отображает ошибку "Пароли не совпадают"
- Пользователь может исправить подтверждение пароля

**Постусловия:**

- Новый пользователь зарегистрирован в системе
- Пользователь авторизован
- Пользователь находится на Dashboard

### UC-1.2: Вход в систему

**Описание:** Зарегистрированный пользователь входит в свой аккаунт.

**Актор:** Зарегистрированный пользователь

**Предусловия:**

- Пользователь не авторизован
- Пользователь имеет зарегистрированный аккаунт

**Основной поток:**

1. Пользователь открывает страницу входа
2. Пользователь вводит Email
3. Пользователь вводит Пароль
4. Пользователь нажимает кнопку "Войти"
5. Система валидирует учетные данные
6. Система проверяет, существует ли пользователь с таким email
7. Система проверяет корректность пароля
8. Система генерирует JWT токен
9. Система сохраняет токен в браузере пользователя
10. Пользователь перенаправляется на Dashboard

**Альтернативные потоки:**

**A1:** Email не найден

- Система отображает ошибку "Неверный email или пароль"
- Пользователь может ввести другой email или перейти на страницу регистрации

**A2:** Неверный пароль

- Система отображает ошибку "Неверный email или пароль"
- Пользователь может ввести пароль заново или использовать ссылку "Забыли пароль?"

**A3:** Валидация формы не пройдена

- Система отображает ошибки валидации для соответствующих полей
- Пользователь может исправить ошибки и попробовать снова

**Постусловия:**

- Пользователь авторизован
- Пользователь находится на Dashboard

### UC-1.3: Выход из системы

**Описание:** Авторизованный пользователь выходит из своего аккаунта.

**Актор:** Авторизованный пользователь

**Предусловия:**

- Пользователь авторизован
- Пользователь находится в любом разделе системы

**Основной поток:**

1. Пользователь нажимает на меню пользователя в правом верхнем углу
2. Пользователь выбирает пункт "Выйти"
3. Система удаляет JWT токен из браузера
4. Система перенаправляет пользователя на страницу входа

**Постусловия:**

- Пользователь не авторизован
- Пользователь находится на странице входа
- JWT токен удален

### UC-1.4: Восстановление пароля

**Описание:** Пользователь забыл пароль и хочет его восстановить.

**Актор:** Зарегистрированный пользователь

**Предусловия:**

- Пользователь не авторизован
- Пользователь имеет зарегистрированный аккаунт

**Основной поток:**

1. Пользователь открывает страницу входа
2. Пользователь нажимает ссылку "Забыли пароль?"
3. Пользователь перенаправляется на страницу восстановления пароля
4. Пользователь вводит свой Email
5. Пользователь нажимает кнопку "Отправить"
6. Система проверяет существование пользователя с таким email
7. Система генерирует токен сброса пароля
8. Система отправляет email с ссылкой для сброса пароля
9. Система отображает сообщение "Письмо отправлено на ваш email"
10. Пользователь открывает письмо в своей почте
11. Пользователь переходит по ссылке в письме
12. Пользователь перенаправляется на страницу сброса пароля
13. Пользователь вводит новый пароль
14. Пользователь подтверждает новый пароль
15. Пользователь нажимает кнопку "Сбросить пароль"
16. Система проверяет валидность токена
17. Система обновляет пароль пользователя
18. Система отображает сообщение об успешном сбросе пароля
19. Пользователь перенаправляется на страницу входа

**Альтернативные потоки:**

**A1:** Email не найден

- Система отображает сообщение "Если аккаунт существует, письмо будет отправлено" (безопасность)
- Пользователь может проверить правильность email или зарегистрироваться

**A2:** Токен истек или невалиден

- Система отображает ошибку "Ссылка недействительна или истекла"
- Пользователь может запросить новую ссылку для сброса пароля

**A3:** Пароли не совпадают

- Система отображает ошибку "Пароли не совпадают"
- Пользователь может исправить подтверждение пароля

**Постусловия:**

- Пароль пользователя обновлен
- Пользователь может войти с новым паролем

## 2. Подключение социальных сетей

### UC-2.1: Подключение аккаунта VK

**Описание:** Пользователь подключает свой аккаунт ВКонтакте к платформе для дальнейшей работы с задачами.

**Актор:** Авторизованный пользователь

**Предусловия:**

- Пользователь авторизован
- Пользователь находится на Dashboard
- Пользователь имеет аккаунт ВКонтакте

**Основной поток:**

1. Пользователь видит раздел "Подключенные социальные сети" на Dashboard
2. Пользователь видит карточку "ВКонтакте" со статусом "Не подключено"
3. Пользователь нажимает кнопку "Подключить" на карточке VK
4. Система открывает модальное окно с информацией о запрашиваемых правах доступа
5. Пользователь нажимает кнопку "Продолжить" в модальном окне
6. Система перенаправляет пользователя на страницу авторизации VK OAuth
7. Пользователь авторизуется в VK (если не авторизован)
8. Пользователь предоставляет необходимые права доступа приложению
9. VK перенаправляет пользователя обратно на платформу с authorization code
10. Система обменивает authorization code на access token и refresh token
11. Система получает информацию о пользователе из VK API
12. Система сохраняет токены и информацию об аккаунте в базе данных
13. Система отображает сообщение об успешном подключении
14. Карточка VK обновляется со статусом "Подключено" и информацией об аккаунте
15. Подключенный аккаунт отображается в списке подключенных аккаунтов

**Альтернативные потоки:**

**A1:** Пользователь отменил авторизацию в VK

- Пользователь остается на Dashboard
- Карточка VK остается со статусом "Не подключено"
- Система не показывает сообщений об ошибке

**A2:** Ошибка авторизации OAuth

- Система отображает ошибку "Не удалось подключить аккаунт. Попробуйте еще раз"
- Пользователь может попробовать подключить аккаунт снова

**A3:** Аккаунт уже подключен

- Система отображает информацию об уже подключенном аккаунте
- Пользователь может отключить аккаунт или подключить другой

**A4:** Недостаточно прав доступа

- Система отображает ошибку "Необходимо предоставить все запрашиваемые права доступа"
- Пользователь может попробовать подключить аккаунт снова и предоставить все права

**Постусловия:**

- Аккаунт VK подключен к платформе
- Токены доступа сохранены в системе
- Аккаунт отображается в списке подключенных аккаунтов
- Пользователь может создавать задачи для этого аккаунта

### UC-2.5: Отключение аккаунта социальной сети

**Описание:** Пользователь отключает подключенный аккаунт социальной сети.

**Актор:** Авторизованный пользователь

**Предусловия:**

- Пользователь авторизован
- У пользователя есть подключенный аккаунт

**Основной поток:**

1. Пользователь открывает список подключенных аккаунтов
2. Пользователь нажимает на кнопку "Отключить" на карточке аккаунта
3. Система отображает диалог подтверждения:
   - "Вы уверены, что хотите отключить этот аккаунт?"
   - "Все невыполненные задачи для этого аккаунта будут отменены"
   - Кнопки "Отменить" и "Отключить"
4. Пользователь подтверждает отключение
5. Система проверяет наличие активных задач для этого аккаунта
6. Система отменяет все активные задачи для этого аккаунта (если есть)
7. Система удаляет токены доступа из базы данных
8. Система удаляет информацию об аккаунте
9. Система отображает сообщение "Аккаунт успешно отключен"
10. Аккаунт удаляется из списка подключенных аккаунтов
11. Карточка социальной сети обновляется со статусом "Не подключено"

**Альтернативные потоки:**

**A1:** Пользователь отменил отключение

- Диалог закрывается
- Аккаунт остается подключенным

**A2:** Ошибка при отключении

- Система отображает ошибку "Не удалось отключить аккаунт. Попробуйте еще раз"
- Аккаунт остается подключенным

**Постусловия:**

- Аккаунт отключен
- Токены доступа удалены
- Аккаунт больше не отображается в списке подключенных
- Все активные задачи для этого аккаунта отменены

### UC-2.6: Обновление токена доступа (Refresh Token)

**Описание:** Система автоматически обновляет истекший токен доступа для подключенного аккаунта.

**Актор:** Система (автоматический процесс)

**Предусловия:**

- У пользователя есть подключенный аккаунт
- Токен доступа истек или скоро истечет

**Основной поток:**

1. Система определяет, что токен доступа истек или скоро истечет
2. Система использует refresh token для получения нового access token
3. Система отправляет запрос к API социальной сети (VK/OK) с refresh token
4. Система получает новый access token и, возможно, новый refresh token
5. Система обновляет токены в базе данных
6. Пользователь не замечает процесса (прозрачно для пользователя)

- Если все попытки неудачны, система помечает аккаунт как неактивный

**Постусловия:**

- Токены обновлены
- Аккаунт остается активным
- Пользователь может продолжать использовать аккаунт

### UC-2.7: Подключение нескольких аккаунтов одной социальной сети

**Описание:** Пользователь подключает несколько аккаунтов одной социальной сети (например, личный аккаунт и аккаунт сообщества VK).

**Актор:** Авторизованный пользователь

**Предусловия:**

- У пользователя уже есть один подключенный аккаунт VK или OK

6. Пользователь авторизуется с другим аккаунтом
7. Система сохраняет новый аккаунт как отдельную запись
8. Оба аккаунта отображаются в списке подключенных аккаунтов

**Примечание:** Этот сценарий может быть реализован в будущих версиях. В первой версии может быть ограничение - один аккаунт на социальную сеть.

**A2:** У выбранного аккаунта нет доступных сообществ

- Система отображает предупреждение "У выбранного аккаунта нет доступных сообществ"

- Пользователь может исправить ошибки и попробовать снова

**A4:** Ошибка при создании задачи

### UC-4.6: Экспорт результатов очистки от ботов

**Описание:** Пользователь экспортирует результаты анализа в файл для дальнейшего использования.

**Актор:** Авторизованный пользователь

**Предусловия:**

- Пользователь авторизован
- Пользователь находится на странице результатов задачи очистки от ботов
- Задача завершена и есть результаты

**Основной поток:**

1. Пользователь нажимает кнопку "Экспорт результатов"
2. Система отображает меню выбора формата экспорта:
   - CSV
   - Excel (XLSX)
   - PDF
   - JSON
3. Пользователь выбирает желаемый формат
4. Система предлагает выбрать данные для экспорта:
   - Все результаты
   - Только боты (фильтр по Bot Score >= порога)
   - Текущий фильтр (если применен)
5. Пользователь выбирает опцию экспорта
6. Система генерирует файл с результатами:

**A2:** Ошибка при экспорте

- Пользователь может попробовать снова или выбрать другой формат

### UC-4.7: Массовое удаление ботов из сообщества

**Описание:** Пользователь массово удаляет найденных ботов из сообщества в социальной сети.

**Актор:** Авторизованный пользователь

**Предусловия:**

- Пользователь авторизован
- Пользователь находится на странице результатов задачи очистки от ботов
- Задача завершена и есть найденные боты
- У пользователя есть права администратора/модератора в сообществе

**Основной поток:**

1. Пользователь просматривает список найденных ботов
2. Пользователь может выбрать ботов для удаления:
   - Выбрать всех ботов (чекбокс "Выбрать все")
   - Выбрать отдельных ботов (чекбоксы рядом с каждой строкой)
   - Выбрать ботов по фильтру (например, все с Bot Score >= 0.9)
3. Пользователь нажимает кнопку "Удалить выбранных"
4. Система отображает диалог подтверждения:
   - "Вы уверены, что хотите удалить {количество} пользователей из сообщества?"
   - "Это действие необратимо"
   - Список выбранных пользователей (или количество)
   - Кнопки "Отменить" и "Подтвердить удаление"
5. Пользователь подтверждает удаление
6. Система проверяет права пользователя на удаление участников
7. Система последовательно удаляет каждого выбранного пользователя через API социальной сети
8. Система отображает прогресс удаления (X из Y удалено)
9. Система отмечает успешно удаленных пользователей в таблице
10. Система отмечает пользователей, которых не удалось удалить (с указанием причины)
11. После завершения система отображает итоговую статистику:
    - Успешно удалено: X
    - Ошибок: Y
12. Таблица результатов обновляется с отметками о статусе удаления

**Альтернативные потоки:**

**A1:** Пользователь отменил удаление

- Диалог закрывается
- Никто не удален

**A2:** Нет прав на удаление

- Система отображает ошибку "У вас нет прав для удаления участников из этого сообщества"
- Удаление не выполняется

**A3:** Ошибка при удалении некоторых пользователей

- Система продолжает удаление остальных
- В таблице отмечаются ошибки для каждого неудачного случая
- Пользователь видит причины ошибок (нет прав, пользователь уже удален, и т.д.)

**A4:** Превышен лимит API

- Система приостанавливает удаление
- Система отображает сообщение "Превышен лимит API. Удаление будет продолжено автоматически через некоторое время"
- Оставшиеся удаления выполняются в фоновом режиме

**Постусловия:**

- Выбранные боты удалены из сообщества (если были права)
- Таблица результатов обновлена со статусами удаления
- Пользователь видит итоговую статистику

**Примечание:** Этот функционал может быть реализован в будущих версиях. В первой версии может быть доступен только просмотр результатов без массового удаления.

## 6. Обработка ошибок и edge cases

### UC-6.1: Обработка ошибок API социальных сетей

**Описание:** Система обрабатывает ошибки при взаимодействии с API VK или Одноклассников.

**Актор:** Система / Авторизованный пользователь

**Сценарии:**

**Сценарий 1: Rate Limiting (Превышен лимит запросов)**

1. Система отправляет запрос к API социальной сети
2. API возвращает ошибку "Too Many Requests" (429)
3. Система определяет, что это rate limiting
4. Система использует exponential backoff для повторных попыток
5. Система ставит задачу в очередь на повторное выполнение
6. Пользователь видит уведомление "Задача приостановлена из-за лимита API. Повтор через X минут"
7. Задача автоматически возобновляется после истечения лимита

**Сценарий 2: Недействительный токен**

1. Система отправляет запрос с токеном доступа
2. API возвращает ошибку "Invalid token" (401)
3. Система пытается обновить токен через refresh token (UC-2.6)
4. Если refresh token валиден - токен обновлен, запрос повторяется
5. Если refresh token невалиден - аккаунт помечается как неактивный, пользователь уведомляется

**Сценарий 3: Недостаточно прав доступа**

1. Система пытается выполнить действие, требующее определенных прав
2. API возвращает ошибку "Insufficient permissions" (403)
3. Система отображает пользователю ошибку "У вас нет прав для выполнения этого действия"
4. Система предлагает пользователю проверить права доступа в настройках социальной сети

**Сценарий 4: Сервис недоступен**

1. Система отправляет запрос к API
2. API не отвечает или возвращает ошибку 500/503
3. Система использует circuit breaker pattern
4. Система повторяет запрос с retry logic (Polly)
5. Если все попытки неудачны - задача помечается как Failed
6. Пользователь видит ошибку "Сервис временно недоступен. Попробуйте позже"

---

### UC-6.2: Обработка длительных задач

**Описание:** Система обрабатывает задачи, выполнение которых занимает длительное время (часы или дни).

**Актор:** Система

**Основной поток:**

1. Задача создается и начинает выполняться
2. Система сохраняет промежуточные результаты в базе данных
3. Система периодически обновляет прогресс задачи
4. Если задача выполняется очень долго:
   - Система сохраняет checkpoint (точку восстановления)
   - Система может приостановить задачу и возобновить позже
   - Система уведомляет пользователя о длительном выполнении
5. Пользователь может закрыть браузер - задача продолжит выполняться
6. При следующем входе пользователь видит прогресс задачи

**Альтернативные потоки:**

**A1:** Ошибка во время выполнения длительной задачи

- Система сохраняет checkpoint перед ошибкой
- Задача помечается как Failed
- Пользователь может возобновить задачу с последнего checkpoint (если возможно)

---

### UC-6.3: Обработка отключенного аккаунта во время выполнения задачи

**Описание:** Пользователь отключает аккаунт социальной сети, пока задача для этого аккаунта выполняется.

**Актор:** Авторизованный пользователь / Система

**Основной поток:**

1. У пользователя выполняется задача для аккаунта VK
2. Пользователь отключает аккаунт VK (UC-2.5)
3. Система определяет, что есть активные задачи для этого аккаунта
4. Система автоматически отменяет все активные задачи для этого аккаунта
5. Система отображает уведомление "Аккаунт отключен. Все активные задачи для этого аккаунта отменены"
6. Пользователь видит отмененные задачи в списке задач

---

### UC-6.4: Обработка большого объема данных

**Описание:** Система обрабатывает задачи с очень большим количеством данных (например, сообщество с миллионами подписчиков).

**Актор:** Система

**Основной поток:**

1. Задача создается для сообщества с большим количеством участников
2. Система определяет, что данных очень много
3. Система разбивает обработку на батчи (chunks)
4. Система обрабатывает каждый батч отдельно
5. Система сохраняет промежуточные результаты
6. Система обновляет прогресс после каждого батча
7. Пользователь видит постепенное обновление прогресса
8. После обработки всех батчей результаты объединяются

**Альтернативные потоки:**

**A1:** Один из батчей провалился

- Система продолжает обработку остальных батчей
- Неудачный батч помечается для повторной обработки
- После завершения успешных батчей система повторяет неудачные

---

### UC-6.5: Одновременное выполнение нескольких задач

**Описание:** Пользователь создает несколько задач одновременно, система корректно их обрабатывает.

**Актор:** Авторизованный пользователь / Система

**Основной поток:**

1. Пользователь создает первую задачу очистки от ботов
2. Пользователь сразу создает вторую задачу для другого сообщества
3. Обе задачи ставятся в очередь RabbitMQ
4. Система обрабатывает задачи параллельно (если есть ресурсы) или последовательно
5. Каждая задача выполняется независимо
6. Пользователь видит прогресс каждой задачи отдельно

**Альтернативные потоки:**

**A1:** Недостаточно ресурсов для параллельной обработки

- Задачи обрабатываются последовательно
- Вторая задача ждет завершения первой (или освобождения ресурсов)

---

### UC-6.6: Восстановление после сбоя системы

**Описание:** Система восстанавливает работу после сбоя или перезапуска.

**Актор:** Система

**Основной поток:**

1. Система перезапускается (планово или из-за сбоя)
2. Система проверяет базу данных на наличие активных задач (InProgress)
3. Система проверяет RabbitMQ на наличие незавершенных сообщений
4. Система определяет, какие задачи были прерваны
5. Система проверяет checkpoint'ы задач
6. Система возобновляет выполнение задач с последнего checkpoint (если возможно)
7. Если checkpoint нет - задачи помечаются как Failed с комментарием "Прервано из-за сбоя системы"
